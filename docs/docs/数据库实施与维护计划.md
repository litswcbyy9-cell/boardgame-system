# 数据库实施与维护计划

## 一、实施步骤（上线顺序）

1. **环境**：安装 MySQL 8.x（`utf8mb4` / `utf8mb4_unicode_ci`），开放应用服务器访问端口（默认 3306）。
2. **初始化**：按编号执行 `db/init` 下 SQL（或 Docker 挂载自动执行）：
   - `01_schema.sql`：表、约束、索引、触发器；
   - `02_procedures.sql`：业务与统计存储过程；
   - `03_seed.sql`：基础字典与示例数据；
   - `04_views.sql`：查询视图；
   - `05_bulk_seed.sql`：大量测试数据（生产环境请删除或跳过）；
   - `06_security_grants.sql`：最小权限账号（可选，按环境调整）。
3. **账号**：开发可使用全权限 `boardgame` 用户；生产建议应用使用 `bg_app`、报表只读使用 `bg_readonly`（见 `06_security_grants.sql`），**部署后立即修改默认口令**。
4. **应用连接池**：Node 端使用连接池（已实现），限制 `connectionLimit`，避免拖垮实例。
5. **前端静态资源**：`npm run build -w web` 生成 `web/dist`，由 Nginx 或 CDN 托管；API 独立域名或同域 `/api` 反代。

## 二、完整性保障

- **实体完整性**：所有表人工主键或单列主键；`AUTO_INCREMENT` 由 MySQL 维护。
- **参照完整性**：外键 `ON DELETE` 策略按表语义设置（如 `game_records` 随 `play_sessions` 级联删除）。
- **域完整性**：枚举状态、`CHECK` 时间窗与金额上界、非空约束。
- **业务一致性**：关键状态（桌位占用/释放、排行榜聚合）由 **触发器 + 存储过程** 双保险，减少应用漏写。

## 三、备份与恢复

| 类型 | 建议 | 说明 |
|------|------|------|
| 全量 | `mysqldump --single-transaction --routines --triggers boardgame > backup.sql` | 包含过程/触发器/视图 |
| 增量 | 开启 **binlog**，结合按时间点恢复（PITR） | 生产推荐 |
| 恢复 | `mysql boardgame < backup.sql` | 先停写或维护窗口 |

## 四、监控与调优

- 监控 **QPS、慢查询、连接数、InnoDB 缓冲池命中率**。
- 对 `reservations` 时间范围查询、`play_sessions` 按日统计，依赖 `01_schema` 中组合索引；若业务 SQL 变化，用 `EXPLAIN` 复核执行计划。
- 大批量导入后执行 `ANALYZE TABLE` 更新统计信息（可选）。

## 五、版本演进（结构变更）

1. 新增迁移脚本 `db/migrations/YYYYMMDD_描述.sql`（人工或 CI 执行），**禁止**仅依赖图形界面改生产库。
2. 涉及触发器/过程/视图时，脚本内显式 `DROP ... IF EXISTS` 再 `CREATE`，保证可重复执行与可追溯。

## 六、安全维护

- 最小权限、分账号、强口令、TLS 连接（云厂商 RDS 一键配置）。
- 定期审计 `mysql.user` 与 `information_schema.USER_PRIVILEGES`。
- `06_security_grants.sql` 仅为课堂/演示模板，生产需与运维规范对齐。

## 七、云端容器化部署（示例）

在已安装 Docker 的云主机上，于**仓库根目录**执行：

```bash
docker compose -f deploy/docker-compose.yml up -d --build
```

浏览器访问 **`http://服务器IP:8080`**（Nginx 反代 `/api` 至 Node）。首次启动 MySQL 会执行 `db/init` 全部脚本（含 `05_bulk_seed` 大量数据，耗时可能数分钟）。生产环境建议：MySQL 使用云 RDS、删除或跳过 `05_bulk_seed.sql`、改写 `06_security_grants.sql` 中的口令与来源网段。
