# 骰子猫桌游馆管理系统演示录像台本

建议视频长度：8 到 12 分钟。

建议录制方式：先打开录屏软件，再运行 `scripts/start-demo-recording.ps1`。这样画面会从项目构建、后端启动、浏览器打开开始，符合“从项目运行开始录制”的要求。

## 录制前准备

1. 打开麦克风，确认系统录屏能录到声音。
2. 关闭不相关窗口，避免泄露浏览器标签、文件路径或密钥。
3. 不要展示 `server/.env` 的完整内容，尤其不要展示大模型 API Key。
4. 建议浏览器缩放设为 100%，窗口分辨率 1440x900 或 1920x1080。
5. 准备好演示账号：`appleadmin / apple123`。

## 第一段：启动项目（约 1 分钟）

画面：

- PowerShell 位于项目根目录 `E:\boardgame-system`。
- 执行：

```powershell
powershell -ExecutionPolicy Bypass -File .\scripts\start-demo-recording.ps1
```

旁白：

> 现在从项目启动开始演示。这个系统是骰子猫桌游馆的单店经营管理系统，前端使用 Vite、Tailwind 和 DaisyUI，后端是 Node.js Express，数据库是 MySQL。启动时会先构建前端，然后由 Express 同时提供 API 和静态页面。

画面重点：

- 构建成功。
- 后端服务启动在 `http://localhost:8788`。
- Chrome 自动打开顾客预约页。

## 第二段：顾客预约页（约 2 分钟）

地址：

`http://localhost:8788/#/customer`

操作：

1. 展示首页视觉、桌游馆定位和预约卡片。
2. 输入顾客姓名、手机号、人数、到店时间、离店时间。
3. 点击匹配桌位。
4. 展示右侧店内桌游目录。
5. 打开顾客 AI 客服，问：“4个人，90分钟，想玩策略但不要太难，有什么推荐？”

旁白：

> 这是顾客公开预约页，不需要登录后台。顾客可以填写人数和时间，系统会自动匹配可用桌位。右侧是店内桌游目录，这些数据来自后台维护的真实桌游数据。这里还接了一个顾客 AI 客服，它会基于店内桌游目录给出推荐，而不是随便编一个不存在的游戏。

强调点：

- 顾客端不应出现后台入口。
- 日期和离店时间输入框不要被挤压。
- 桌游数量不是 0，能看到馆内桌游。

## 第三段：后台登录与首页（约 1 分钟）

操作：

1. 进入 `http://localhost:8788/#/login`。
2. 使用 `appleadmin / apple123` 登录。
3. 展示后台首页。
4. 点一下常驻 AI 桌宠。

旁白：

> 接下来进入后台。后台是给店员和管理员使用的，登录后可以看到今日经营概览。右下角这个常驻的 AI 桌宠可以随时唤起，用来问经营数据、活动建议或者桌游推荐。

## 第四段：桌位运营闭环（约 2 分钟）

页面：

桌位运营。

操作：

1. 展示桌位平面图：空闲、已预约、使用中。
2. 选择一个空闲桌位。
3. 演示创建预约或现场开台。
4. 如果已有进行中桌位，演示结算。

旁白：

> 桌位运营是门店现场使用频率最高的模块。这里能看到每张桌子的实时状态，可以创建预约、办理到店、现场开台，也可以结束桌位使用并结算。结算不是简单地改状态，它会同时形成订单、写入会员消费、计算积分，并更新会员等级。

## 第五段：会员、优惠券与订单（约 1.5 分钟）

页面：

会员管理、优惠券、订单结算。

操作：

1. 在会员页搜索会员。
2. 展示余额、积分、等级、历史预约。
3. 切到优惠券页，展示券种。
4. 切到订单页，展示订单统计。

旁白：

> 会员模块保留了桌游馆最常用的储值、消费、积分和等级。优惠券模块用于基础营销，订单模块记录每次结算形成的收入。后续如果要接微信或支付宝，这里就是支付回调之后落账的位置。

## 第六段：桌游目录与 AI 文案（约 1.5 分钟）

页面：

桌游管理。

操作：

1. 搜索一款桌游。
2. 打开编辑弹窗。
3. 展示字段：人数、时长、难度、分类、描述、封面。
4. 点击 AI 生成描述按钮，或展示已有描述。

旁白：

> 桌游目录不是一个静态列表，它会被顾客页、AI 推荐、租借和战绩模块共同使用。管理员可以自由添加桌游、修改描述、维护封面和难度。AI 可以帮助生成介绍文案，减少维护成本。

## 第七段：租借服务与战绩计分（约 1.5 分钟）

页面：

租借服务、战绩排行。

操作：

1. 打开租借服务，展示副本状态和借出记录。
2. 演示可借、借出、逾期、遗失统计。
3. 打开排行榜，切换胜率 / ELO。

旁白：

> 这里扩展了桌游馆的两个经营场景。租借服务把每一盒桌游当成资产管理，可以记录副本、押金、借出和归还。战绩模块支持多人对局和 ELO 计分，适合做会员赛季、排行榜和线下活动。

## 第八段：AI 经营助手（约 1 分钟）

页面：

AI 助手。

推荐提问：

```text
请根据当前桌游目录和经营数据，帮我设计一个周末四人桌游活动，并推荐 3 款适合新手的游戏。
```

旁白：

> 最后是 AI 经营助手。它接的是 OpenAI 兼容接口，我这里可以使用已经配置的大模型。它会读取系统里的经营摘要和桌游目录，所以可以回答和门店经营相关的问题。后续它可以继续扩展成日报、活动策划、顾客推荐和自动运营建议。

## 结束语（约 30 秒）

旁白：

> 总结一下，这个系统现在已经跑通了桌游馆单店经营的主流程：顾客预约、桌位运营、会员储值、结算订单、桌游目录、租借服务、战绩计分和 AI 助手。支付、短信和小程序属于外部资源依赖，暂时没有接入；但门店内部闭环已经可以完整演示和继续打磨。

## 精简版 3 分钟旁白

如果只录短视频，可以用下面版本：

> 这是骰子猫桌游馆管理系统。现在从项目启动开始，前端构建后由 Express 同时提供页面和 API，浏览器打开的是顾客预约页。
>
> 顾客不用登录后台，可以直接选择人数和时间，系统会匹配可用桌位。右侧的桌游目录来自后台真实数据，AI 客服会结合这些桌游给顾客推荐适合的游戏。
>
> 后台登录后，店员可以处理桌位、预约、会员和结算。桌位结算会自动形成订单，更新会员消费、积分和等级。
>
> 管理员可以维护桌游目录、修改描述和封面，也可以用 AI 生成介绍文案。系统还加入了桌游租借服务，把每一盒桌游作为资产管理，并支持借出、归还和遗失状态。
>
> 战绩模块支持多人对局记录和 ELO 排名，可以用于会员赛季和活动运营。AI 助手则作为常驻经营助手，能根据门店数据和桌游目录回答问题、推荐游戏、生成活动方案。
>
> 目前支付、短信和小程序这些外部依赖暂未接入，但单店经营闭环已经完整跑通。
